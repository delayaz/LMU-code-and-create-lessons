if, elif, else — sorting people into tiers

An if statement runs a block of code only when something is true. Add elif for more options, and else to catch everything left over. Let's build a ticket counter for a movie theater that prices tickets by age.

main.py
age = 15

if age < 5:
    price = 0
elif age < 18:
    price = 8
elif age < 65:
    price = 12
else:
    price = 9

print(f"Age {age}: ${price} ticket")
$ Age 15: $8 ticket

Notice — Python checks each condition top to bottom, and stops at the first one that's true. That's why these are written in increasing order (under 5, then under 18, then under 65) — order matters here.

Pick your own venue — a movie theater, a concert, an arcade, a theme park — and decide your own age tiers and prices. Use input() to ask for age, convert it to a number, then if/elif/else to print the ticket price.

  • Remember: input() always returns text — wrap it in int(...) to turn it into a real number you can compare with < or >=.
  • What goes wrong if you write your conditions in decreasing order instead (65 first, then 18, then 5)? Nothing breaks syntactically, but you'd need to flip the comparisons to >= instead of < — mixing the two orderings up is a common source of silent bugs.
!Common snag: forgetting int() around the input, so Python tries to compare text to a number and throws a TypeError. Also watch indentation — every line inside an if block needs the exact same indent.

and, or, not — now build the bouncer

Ticket pricing only needed one piece of info. A real bouncer needs to check several things at once. and needs both sides true. or needs just one. not flips true to false.

main.py
age = 17
has_ticket = True
has_id = False

if age >= 18 and has_ticket:
    print("Come on in.")
elif age >= 13 and not has_ticket:
    print("You need a ticket first.")
else:
    print("Sorry, you can't come in.")
$ You need a ticket first.

!Common snag: mixing up when you need and vs or. "Let in if 18+ and has ticket" is strict — both required. "Let in if on the list or has a ticket" is generous — either one works.

● AI Segment #1

How do tools like ChatGPT and Claude actually work?

You're going to be tempted to use AI tools the rest of this course — so let's set the norms now, before any problem comes up. A quick mental model: these tools predict the next most likely word, over and over, based on patterns in huge amounts of text. They're not "thinking" the way you are, and they're not always right.

Good use

Asking AI to explain an error message in plain language, or to suggest a few different directions for a project idea you're stuck on.

Lazy use

Asking AI to just write the whole thing for you — you'll have code that runs, but you won't understand it, and you can't fix it when it breaks.

Rule of thumb for this course: AI is a tutor, not a typist. If you can't explain what a line of code does after asking AI for help with it, slow down and ask a human instead — Delaram, Lauren, or a neighbor.
