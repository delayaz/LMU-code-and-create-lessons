example — GUI (tkinter)

Same Quiz, Real Window

Identical questions, but with a dark-themed window, clickable buttons, and colour feedback. Shows what adding a GUI on top of existing logic looks like.

def check_answer(idx):
    correct = state["questions"][state["current"]]["answer"]
    if idx == correct:
        opt_btns[idx].config(bg=GREEN, fg=BG)   # turn green
        feedback_lbl.config(text="✓ Correct!")
    else:
        opt_btns[idx].config(bg=RED, fg=TEXT)    # turn red
        opt_btns[correct].config(bg=GREEN, fg=BG)

tkinter is the same Python you already know. .config() lets you update a button’s color or text after it’s been created — that’s how visual feedback works.

⬡ local Python only
example — pygame

Catch!

Arcade-style falling-object game with a moveable paddle, difficulty scaling, three object types, and a start/game-over screen.

# Falling objects are a list of dicts — same as Day 5
OBJECT_TYPES = [
    {"label": "good", "color": GREEN, "speed": 3.0, "points": 1},
    {"label": "good", "color": BLUE,  "speed": 5.0, "points": 2},
    {"label": "bad",  "color": RED,   "speed": 3.5, "points": 0},
]

To add a new type of falling object, add one more dictionary to this list. That’s it — the rest of the game picks it up automatically.

Track 3 — Interactive Story

Choose Your Adventure

Each scene is its own function. One scene asks a question, then calls the next scene based on the answer. This keeps branches readable and easy to extend one scene at a time.

def opening():
    # TODO: Replace this with your real opening.
    print("You wake up in a mysterious forest.")
    choice = input("Go LEFT or RIGHT? ").lower()
    if   choice == "left":  scene_left()
    elif choice == "right": scene_right()
First thing to try: rewrite opening() with your real first scene and first choice. Once that works, write scene_left() and scene_right() — then keep branching.

One habit
Reminder

Run it constantly — don’t save up bugs

The single most useful habit in a short project: run your code after every 3–4 lines. Not after you finish a function. Not after a whole section. Every few lines. Bugs caught immediately are usually one-line fixes — bugs saved up for 20 minutes are sometimes project-ending.

the workflow that works
# Step 1: write 2-3 lines
questions = []
questions.append({"question": "What's your first real question?", "answer": "..."})

# Step 2: run it — does it print without errors? Good.
print(questions[0])

# Step 3: add the next piece — the ask function
def ask_question(q):
    print(q["question"])
    answer = input("Your answer: ")
    return answer == q["answer"]

# Step 4: run it again — call the function on one question
print(ask_question(questions[0]))

# Step 5: once that works, add the loop. Never the other way around.

The best project is the one you actually finish.

Keep it small enough to complete, interesting enough to keep you going. A tight, finished program you built yourself beats an ambitious half-finished one every time.

Ask for help the moment something doesn’t run — don’t troubleshoot alone for more than a few minutes.
